# v1.0 Multi-Tenant Foundation Plan

## Overview

The v1.0 milestone moves Claudriel from a prototype into a multi-tenant production foundation. The work is organized to make deployment behavior reliable first, define tenant and workspace boundaries clearly, validate those boundaries with smoke coverage, enforce them in routing, harden production access, and then round out the workflow support that helps future execution stay consistent.

## Phase 1: Deployment Reliability (#90)

### Objectives

- Make the sidecar build and deployment path reliable across CI and production promotion.
- Reduce rollout risk for the tenant boundary and routing changes that follow.

### Required Work

- Audit the current GitHub Actions and sidecar deployment path.
- Remove failure points that make sidecar promotion inconsistent across environments.
- Verify that the canonical repo-managed deployment path is stable.

### Expected Artifacts

- Updated deployment workflow configuration.
- Verified sidecar build and promotion path.
- Notes or checks that document the stable deployment path.

### Completion Signals

- Sidecar deployment succeeds consistently through the intended GitHub Actions flow.
- Production promotion no longer depends on manual recovery or ad hoc steps.
- Follow-on architectural and hardening work can rely on the deployment path.

### Dependencies

- No hard prerequisite inside v1.0.
- Must be in place before or alongside Phase 2 and before Phase 5.

## Phase 2: Boundary Definition (#91)

### Objectives

- Define the canonical tenant and workspace boundary model for Claudriel.
- Establish the rules that routing, storage, and operational behavior must obey.

### Required Work

- Define tenant and workspace concepts unambiguously.
- Document enforcement points for routing, persistence, jobs, and sidecar behavior.
- Specify ownership and isolation rules that downstream implementation must follow.

### Expected Artifacts

- A completed issue definition in #91 that serves as the architectural anchor.
- Boundary rules that downstream implementation can reference directly.

### Completion Signals

- Tenant and workspace terms are unambiguous.
- Enforcement points and ownership rules are documented.
- Routing implementation work can proceed without guessing at the model.

### Dependencies

- Should be completed after or alongside Phase 1.
- Should be completed after or alongside Phase 3.
- Must complete before Phase 4.

## Phase 3: Smoke Coverage (#92)

### Objectives

- Add production-facing smoke coverage for the daily brief and chat interface.
- Create validation coverage for the tenant and workspace boundaries being defined in v1.0.

### Required Work

- Identify the critical production-facing brief and chat flows that need smoke coverage.
- Add reliable checks for those flows.
- Ensure the coverage is usable while boundary and routing work is landing.

### Expected Artifacts

- Smoke tests or equivalent production-facing checks for daily brief and chat.
- Verification criteria that can be reused during rollout and hardening.

### Completion Signals

- Critical brief and chat flows have repeatable smoke coverage.
- The new boundary model can be validated through production-facing checks.
- Production hardening work can proceed with meaningful validation coverage.

### Dependencies

- No hard prerequisite inside v1.0.
- Should be completed before or alongside Phase 2.
- Should be completed before or alongside Phase 5.
- Makes Phase 4 safer even though Phase 4's hard dependency is Phase 2.

## Phase 4: Routing Enforcement (#94)

### Objectives

- Implement routing and request resolution that obey the tenant and workspace boundary model.
- Carry tenant and workspace context through runtime request handling safely.

### Required Work

- Translate the boundary rules from #91 into routing and request-resolution behavior.
- Enforce tenant and workspace context in the routing layer.
- Validate that routing behavior aligns with the documented boundary model.

### Expected Artifacts

- Tenant-aware routing implementation.
- Request-resolution behavior that respects tenant and workspace context.
- Supporting validation that the routing layer follows the architectural boundary rules.

### Completion Signals

- Routing resolves tenant and workspace context as intended.
- The implementation conforms to the rules defined in #91.
- The change is safe to exercise through the smoke coverage from #92.

### Dependencies

- Hard dependency on Phase 2.
- Safer after Phase 1 and Phase 3.

## Phase 5: Production Hardening (#89)

### Objectives

- Remove production basic auth from the Caddyfile without weakening deployment safety.
- Ensure production access hardening aligns with the stable deployment path and validation coverage.

### Required Work

- Remove or replace the current production basic auth configuration through the canonical deployment path.
- Validate the change using the smoke coverage established for production-facing flows.
- Confirm the change does not create regressions in the deployment workflow.

### Expected Artifacts

- Updated production Caddyfile or deployment-managed config.
- Validation evidence that production remains healthy after auth removal.

### Completion Signals

- Basic auth is removed through the repo-managed path.
- Production remains reachable and validated after rollout.
- The change does not depend on fragile manual intervention.

### Dependencies

- Should land after Phase 1 is at least partially complete.
- Should land after Phase 3 is at least partially complete.

## Phase 6: Workflow Support (#93)

### Objectives

- Strengthen the Codex environment skills and workflow conventions that support the v1.0 program.
- Improve execution consistency without blocking product-critical tenant work.

### Required Work

- Define and maintain the Codex environment skills, repo conventions, and operational boundaries for Claudriel development.
- Align workflow guidance with the multi-tenant foundation work as needed.

### Expected Artifacts

- Updated or clarified workflow guidance and skill support.
- Operational conventions that reduce ambiguity during ongoing v1.0 work.

### Completion Signals

- The repo workflow and environment guidance reflect the current operating model.
- Contributors can execute the v1.0 work with less ambiguity.

### Dependencies

- Independent within v1.0.
- Can run in parallel and remains the lowest product-criticality item.

## Dependency Graph

- `#90 -> #91 -> #94`
- `#92 -> #94`
- `#89` after `#90` and `#92`
- `#93` independent
