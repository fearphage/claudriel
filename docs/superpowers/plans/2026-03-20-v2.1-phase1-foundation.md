```php

use PHPUnit\Framework\TestCase;

final class CliOnFreshDatabaseTest extends TestCase

    {
        // Use a temp SQLite file to simulate fresh DB
        $tempDb = tempnam(sys_get_temp_dir(), 'claudriel_test_') . '.sqlite';

        try {
            $result = shell_exec(
                sprintf(
                    'CLAUDRIEL_DATABASE_PATH=%s php %s/bin/claudriel claudriel:brief 2>&1',

                    escapeshellarg(dirname(__DIR__, 3))
```

### Task 2: Fix CORS blocking admin SPA (#385)

**Problem:** Admin SPA (static Nuxt build) makes `fetch()` calls to `/api/schema/{type}` and `/graphql`. In production, these may be blocked by CORS or missing reverse proxy rules.

**Files:**
- Modify: `Caddyfile` (or Ansible template `waaseyaa-caddyfile.j2`)
- Possibly modify: PHP middleware for CORS headers

- [ ] **Step 1: Reproduce the issue**

Check the current Caddy config for API routing:

```bash
cat Caddyfile | grep -A 5 "api\|graphql\|cors"
```

Check if the admin SPA requests are being served correctly:
```bash
curl -v -H "Origin: https://claudriel.ai" https://claudriel.ai/api/schema/commitment 2>&1 | grep -i "access-control\|cors\|status"
```

- [ ] **Step 2: Identify the fix**

Two possible approaches:
a) **Caddy-level:** Add CORS headers and ensure `/api/*` routes reverse proxy to PHP-FPM
b) **PHP-level:** Add CORS middleware in Waaseyaa

Since admin SPA is served from the same origin in production (static files under `/public/admin/`), this may be a same-origin issue where Caddy isn't routing `/api/*` to PHP-FPM. Check the Caddyfile for `php_fastcgi` matcher scope.

- [ ] **Step 3: Apply the fix**

If it's a Caddy routing issue, ensure the `php_fastcgi` directive covers `/api/*` and `/graphql` paths. The fix may need to go in the Ansible template (`waaseyaa-caddyfile.j2`) per CLAUDE.md gotcha.

If it's a CORS issue (admin on different port during dev), add appropriate headers:
```
header /api/* {
    Access-Control-Allow-Origin *
    Access-Control-Allow-Methods "GET, POST, OPTIONS"
    Access-Control-Allow-Headers "Content-Type, Authorization"
}
```

- [ ] **Step 4: Test the fix**

Verify API calls work from the admin SPA:
```bash
# Dev: check admin can reach API
curl -s http://localhost:8081/api/schema/commitment | head -5
```

- [ ] **Step 5: Commit**

```bash

git commit -m "fix(#385): add CORS/routing for admin SPA API requests"
```

### Task 4: Make daily ceiling check atomic (#326)

**Problem:** `InternalSessionController::continueSession()` loads all tenant sessions, filters by date in PHP, sums turns. TOCTOU race on concurrent requests.

**Files:**
- Modify: `src/Controller/InternalSessionController.php:68-90`
- Test: `tests/Integration/Controller/InternalSessionControllerTest.php`

- [ ] **Step 1: Assess severity**

The code comment says "TOCTOU race exists on concurrent requests; acceptable given single-user tenants." If Claudriel remains single-tenant per user, this is low priority.

Decision: If single-tenant is the design constraint, update the issue with a "wontfix" label and close. If multi-tenant is planned, implement the fix below.

- [ ] **Step 2: Write test for ceiling enforcement**

```php
public function testDailyCeilingPreventsExcessiveTurns(): void
{
    // Create sessions that sum to DAILY_CEILING
    // Attempt continueSession
    // Assert 429 response
}
```

- [ ] **Step 3: Implement atomic check**

Option A (simplest, no schema change): Use a database transaction with `SELECT ... FOR UPDATE` equivalent. Since SQLite doesn't support row locking, use `BEGIN IMMEDIATE` transaction:

```php
// Wrap the ceiling check + increment in a transaction
$this->database->beginTransaction();
try {
    $totalTurnsToday = $this->countTodayTurns();
    if ($totalTurnsToday >= self::DAILY_CEILING) {
        $this->database->rollback();
        return $this->jsonError('Daily turn ceiling reached', 429);
    }
    $session->set('continued_count', $continuedCount);
    $this->sessionRepo->save($session);
    $this->database->commit();
} catch (\Throwable $e) {
    $this->database->rollback();
    throw $e;
}
```

Note: This requires injecting `DatabaseInterface` into the controller. Check if that's feasible with the current DI setup.

Option B (accept trade-off): Close the issue with documentation that single-tenant architecture makes the race benign.
