# v1.0 Smoke Matrix

## Purpose

This document defines the Phase 3 smoke coverage target for issue `#92`. It focuses on the critical daily brief and chat flows that must remain valid while the tenant/workspace boundary model is defined in `#91` and later enforced in `#94`.

## Validation Approach

- Use production-facing unauthenticated HTTP checks or scripted browser/API probes against the deployed application.
- Prefer small deterministic checks that confirm availability, expected status codes, and key response markers.
- Validate the happy path first, then validate that invalid or unsafe requests fail closed.
- Treat this matrix as the baseline for rollout validation during Phase 4 and Phase 5.

## Smoke Test Matrix

| Flow | Surface | Preconditions | Check | Expected result | Phase support |
|---|---|---|---|---|---|
| Daily brief HTML render | `GET /brief` | App deployed and reachable | Request the page in a browser-like client | `200`, HTML renders, no server error | Validates production readiness before auth hardening |
| Daily brief JSON render | `GET /brief` with JSON accept header | App deployed and reachable | Request JSON variant | `200`, JSON response shape remains valid | Protects API-style brief consumers during routing changes |
| Dashboard render with workspace cards | Main dashboard page | App reachable with seeded workspace data | Load dashboard | `200`, page renders, workspace section visible when data exists | Confirms tenant-scoped UI continues to render |
| Chat stream start | Chat entry and stream endpoint | Chat configured with API key or sidecar fallback | Submit a simple chat message and open SSE stream | Stream starts, emits progress or response events, no `500` | Baseline for routing and sidecar-safe changes |
| Chat sidecar fallback behavior | Chat stream with sidecar unavailable | Sidecar intentionally unavailable or health check failing | Submit a chat request | App degrades honestly, no fatal crash, fallback path works | Reduces rollout risk while deployment work evolves |
| Workspace creation through chat local action | Chat local action path | Existing session/message path available | Send `create a workspace named "X"` | Workspace is created, assistant confirms action, no server error | Confirms workspace mutations remain stable before boundary enforcement |
| Workspace deletion through chat local action | Chat local action path | Workspace exists | Send `delete workspace X` | Workspace is deleted or missing state is reported cleanly | Confirms local workspace mutation path remains safe |
| Brief stream endpoint | Brief streaming surface | App reachable | Connect to brief stream | Streaming response starts without server error | Protects the live brief experience during v1.0 work |

## Negative Case Table

| Case | Surface | Check | Expected result | Why it matters |
|---|---|---|---|---|
| Unsupported brief route or bad path | HTTP routing | Request unknown route near brief/dashboard surfaces | `404`, not `500` | Confirms routing changes fail safely |
| Missing chat configuration | Chat | Exercise chat when Anthropic key is absent in a controlled environment | `503` with configuration error, not `500` | Ensures misconfiguration is survivable |
| Sidecar unavailable | Chat | Simulate unhealthy sidecar | Graceful fallback or explicit non-fatal error path | Supports deployment and rollback validation |
| Unknown workspace deletion request | Chat local action | Delete non-existent workspace | Clean assistant response, no crash, no unintended delete | Protects workspace mutation safety |
| Duplicate workspace creation request | Chat local action | Create an existing workspace again | Existing workspace reported, no duplicate side effect | Reduces mutation ambiguity before tenant scoping lands |
| Invalid workspace scope in future tenant-aware flows | Dashboard/chat/api | Use mismatched or unknown workspace identifier once routing enforcement exists | Refuse or return empty scoped result, never cross-scope data | Defines the future safety bar for Phase 4 |
| Deploy regression on brief or chat | Production release validation | Run unauthenticated public brief and chat checks immediately after deploy | Both surfaces stay healthy after release | Supports Phase 5 production hardening |

## Critical Flows Covered Now

- Daily brief HTML and JSON rendering
- Dashboard render with workspace-aware UI context
- Chat streaming startup
- Sidecar fallback behavior
- Workspace create/delete local actions through chat
- Brief streaming surface health

## How This Supports Later Phases

- Phase 4 can use these checks to confirm tenant-aware routing does not break brief or chat entry points.
- Phase 5 can use these public Caddy endpoint checks after basic auth removal to verify production remains healthy without route-specific auth exceptions.
- The negative cases define the fail-closed behavior expected once tenant and workspace enforcement is implemented.
